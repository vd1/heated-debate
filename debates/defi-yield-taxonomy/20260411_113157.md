<details><summary><strong>Topic</strong></summary>

Which decomposition of DeFi yield is most useful for portfolio risk analysis?

## Background

A practitioner taxonomy identifies four "sources" of DeFi yield: farming (incentives), chain staking, arb-fund (liquidation/arbitrage/funding-rate extraction), and RWAs. But these categories mix ontological layers — farming is a funding mechanism, staking is a protocol role, arb-fund is a strategy class, and RWA is an asset origin. They are not peer categories, which causes border disputes (where do LP fees go? lending spreads? MEV?) and hides dependency chains in blended products like sUSDe or Morpho vaults.

## Competing frameworks to evaluate

### Framework A: The 4-Source Folk Taxonomy
Farming | Staking | Arb-fund | RWA
Simple, widely used, but categories answer different questions.

### Framework B: Payer x Mechanism Matrix
- Axis 1 — Who pays: protocol treasury, network users, traders, borrowers, hedgers, offchain borrowers, other participants
- Axis 2 — Economic mechanism: subsidy, validation, market-making, balance-sheet rental, ordering-rights extraction, risk warehousing, offchain pass-through
- Handles blended products by decomposing them into multiple cells

### Framework C: Economic Character Lens (Productive / Extractive / Subsidized / Pass-through)
Heuristic buckets for persistence and failure mode, not mutually pure ontological classes. A single position can exhibit multiple characters (e.g., staking is productive + extractive via MEV).
- Productive: you provide a service someone needs (LP, lending, insurance, staking)
- Extractive: you capture value from information asymmetry / structural inefficiency (MEV, arb, liquidation)
- Subsidized: you receive tokens from a depleting treasury (farming, liquidity mining)
- Pass-through: you wrap offchain cash flows onchain (RWA, T-bills, private credit)

### Framework D: Risk-first taxonomy
Classify not the yield source but the risk you are SHORT to earn it.

Core short-risks (the thing you are being paid to bear):
- Inventory / adverse selection (LP, market making)
- Credit / default (lending, undercollateralized positions)
- Duration / rate risk (fixed-rate lending, maturity mismatch)
- Liquidity / lock-up (staking, vesting)
- Volatility / convexity (options selling, DOVs)
- Governance / dilution (farming, token-incentivized positions)

Universal overlays (present in all DeFi, not the thing you are specifically paid to short):
- Smart contract risk
- Oracle risk
- Legal / regulatory risk

### Framework E: No single taxonomy — use a matrix with multiple projections depending on the question being asked.

## Classification rule

Every framework must classify each test case on three layers:
1. **Payer**: who funds the yield?
2. **Mechanism**: what economic function produces it?
3. **Risk absorbed**: what are you short?

This prevents MEV, LP fees, and lending from collapsing into single-label arguments.

## Concrete test cases the debate must address

Each framework should be evaluated against these hard cases:

1. **Ethena sUSDe**: blends perp funding rates + token incentives. How does each framework decompose it? Does the decomposition reveal risk?
2. **Morpho vault lending**: lending spread where borrower demand is downstream of farming incentives elsewhere. Is the yield "productive" if its payer is subsidized?
3. **Uniswap v3 concentrated LP**: fee income minus adverse selection / impermanent loss. Net yield can be negative. Where does this go?
4. **MEV on Ethereum**: simultaneously validator reward, searcher profit, and user extraction depending on vantage point. Can a framework handle all three perspectives?
5. **Nexus Mutual / DeFi options vaults (DOVs)**: selling risk. Is this productive (service), extractive (mispriced risk), or its own thing?
6. **Term Finance fixed-rate lending**: lending spread, but fixed and maturity-matched. Does the framework capture how this differs from variable-rate lending?
7. **Resolv USR exit shock**: curator behavior affecting borrow rates across markets. How does each framework represent contagion?

## What "most useful" means

The winning framework should:
- **Isolate sustainability**: make it immediately visible whether yield depends on endogenous token emissions or exogenous cash flows
- **Reveal dependency chains**: show when one yield source is downstream of another (lending -> borrower farming -> treasury depletion)
- **Enable risk attribution**: let a portfolio manager answer "what am I short?" for each position
- **Handle blended products**: decompose products that combine multiple yield sources without forcing them into one bucket
- **Separate gross cash-flow source from net strategy return**: frameworks should distinguish where revenue comes from vs what risks and costs consume it (LP fees are gross revenue; net LP yield is after adverse selection, inventory risk, IL, gas, rebalancing; funding-rate carry is gross carry before hedge costs and basis slippage)
- **Be practically usable**: a risk analyst should be able to classify a new vault in < 5 minutes

Creating a fake 'shock transmission leg' violates the unit-of-classification rule.
- The economic counterparty field is mandatory for every leg. An additional 'adverse incidence / extraction channel' annotation is added only when the leg involves extraction (e.g., MEV searcher profit explicitly names traders/protocols as bearing worse execution). This avoids forcing productive yield like lending or insurance into a zero-sum framing.
- For Uniswap v3 concentrated liquidity positions, the primary risk is LVR (loss-versus-rebalancing) / adverse selection plus inventory convexity — not generic 'impermanent loss.' IL is the accounting manifestation that belongs in the drags field, not the core economic risk classification. Arbitrageurs and searchers are the economic counterparties realizing the adverse incidence.
- For MEV (maximal extractable value — profit extracted by reordering, inserting, or censoring blockchain transactions), the decomposition requires linked legs: proposer/validator revenue and searcher/arbitrageur profit must be represented together with explicit notation of the downstream incidence on users and protocols. A single-row treatment hides the extraction.
- Nexus Mutual (DeFi insurance) and DOVs (DeFi Options Vaults that sell options to earn premium) must be classified as separate legs despite both 'warehousing risk,' because their stress surfaces differ fundamentally: Nexus is short correlated default/claims severity while DOVs are short volatility/convexity/gamma.
- For Term Finance-style fixed-rate lending, the primary risk is credit plus exit-liquidity, not duration. Duration risk only materializes if the position is marked to market or exited before maturity. Fixed-vs-floating distinction belongs in the risk and drag fields, not as a separate source category.

**Open Questions:**
- Should the portfolio be aggregated on a single axis (primary compensated short-risk, as in the D-first proposal from Rounds 2-4) or on a 2D grid (base carry driver × dominant failure driver, as proposed in Round 5)? The single-axis approach is simpler but may create false diversification by making structurally similar trades look different. The 2D approach is more honest but harder to operationalize for limits and stress testing.
- For each of the seven reference cases (sUSDe, Morpho vault, Uniswap v3 LP, Ethereum MEV, Nexus Mutual, DOVs, Term Finance, Resolv USR), does removing a supposed fragility overlay materially compress the expected yield? If yes, that overlay is priced risk and must be promoted to a first-class failure driver. This re-test has been proposed but not yet executed.
- How should legs where the dominant PnL driver changes under stress (regime-switch legs) be handled in the portfolio book? Two options were proposed: (a) split the leg into regime-specific rows, or (b) keep one row with a mandatory regime-switch flag. Neither was validated against the reference cases.
- Should 'dual-dominant' failure drivers (where two failure modes are jointly indispensable to the leg) require splitting the leg into sub-legs, or is a dual-dominant annotation sufficient? This matters for cases like sUSDe where both basis/funding rollover failure and venue/custody concentration may be jointly first-order.
- What is the correct closed vocabulary for 'base carry driver'? Round 5 proposed: inventory-convexity, credit, liquidity-basis, volatility, governance-dilution, ordering-rights access, offchain pass-through. What is the correct closed vocabulary for 'dominant failure driver'? Neither list has been validated against all seven cases.

**Best Ideas:**
- Leg-based decomposition protocol: Instead of classifying DeFi products or positions into a single yield category, decompose every position into its constituent cash-flow legs and classify each leg independently. This matters because complex products like Ethena's sUSDe (a stablecoin earning yield from perpetual futures funding rates plus token incentives) or Morpho vaults (curated lending aggregators that may route deposits across multiple lending markets) combine economically distinct income streams that have different payers, different risks, and different sustainability profiles. By classifying legs rather than products, you avoid the ambiguity that plagues every product-level taxonomy. The protocol works by: (1) identifying each distinct income stream, (2) recording its payer chain, mechanism, risk profile, and counterparty in a structured ledger row, and (3) re-aggregating across the portfolio by risk dimensions rather than product labels.
- Two-dimensional portfolio control surface: Rather than aggregating the portfolio on a single risk axis, use a 2D grid where one axis is the 'base carry driver' (the economic reason you are being paid — e.g., credit risk, inventory convexity, liquidity/basis risk, volatility) and the other axis is the 'dominant failure driver' (the structural fragility most likely to cause loss — e.g., venue/custody dependence, allocator discretion, exit liquidity, endogenous-reflexive demand). This matters because in DeFi, unlike traditional finance, the structural fragility is often part of what the market is compensating you for. A single-axis aggregation by compensated risk creates false diversification: trades that look different by their carry driver may share the same dominant failure mode (e.g., multiple positions all dependent on the same custody stack or the same allocator's routing decisions). The 2D view exposes these hidden concentrations.
- Promotion rule for fragility overlays: Any risk initially classified as a 'fragility overlay' (a risk that can break the trade but is not the primary economic reason you are paid) must be promoted to a first-class 'dominant failure driver' whenever removing it would materially compress the expected yield. This matters because DeFi yields frequently embed compensation for structural fragilities that traditional finance would treat as operational risk — custody concentration, smart contract dependence, allocator discretion, venue lock-in. If these are left in metadata while only classical financial risks (credit, liquidity, volatility) populate the main risk taxonomy, the portfolio book systematically understates the concentrations that matter most. The test is concrete: ask 'Would this yield materially compress if this supposed overlay vanished?' If yes, it is priced risk.
- Terminal funding overlay as a hard sustainability field: Instead of using subjective labels like 'productive' or 'extractive' to assess yield sustainability, force every leg to declare its terminal funding source from a closed list of five options: external cash flow (revenue from outside the crypto ecosystem), treasury subsidy (protocol token emissions or treasury drawdowns), user-demand fee (users paying for a genuine service like trading, borrowing, or insurance), zero-sum transfer (one participant's gain is another's loss, as in MEV or arbitrage), or reflexive (the yield depends on continued growth or token price appreciation in a circular way). This matters because sustainability debates in DeFi are notoriously subjective and narrative-driven. A hard field with five options forces precision and makes it immediately visible when a 'productive' lending mechanism is actually one hop away from treasury-subsidized borrower demand.
- Regime-switch detection rule: If the dominant PnL driver of a cash-flow leg changes under stress conditions while the leg itself remains the same (same payer, same mechanism, same gross source), that leg cannot be stably classified in a single risk bucket. It must either be split into regime-specific rows (normal regime vs. stress regime) or carry a mandatory regime-switch flag. This matters because DeFi yield strategies frequently exhibit regime dependence — a carry trade that is compensating you for basis risk in normal conditions may suddenly become a pure exit-liquidity / venue-concentration trade in stress. If the risk book does not flag this, it will show the portfolio as diversified across different risk buckets when in reality multiple positions converge to the same failure mode under stress.

**Concrete Proposals:**
- Implement a portfolio ledger where each row represents one holder cash-flow leg with these mandatory fields: holder cash-flow leg (description of the income stream), immediate payer (who directly pays), terminal funding source (one of: external cash flow / treasury subsidy / user-demand fee / zero-sum transfer / reflexive), mechanism (one of: subsidy / validation / market-making / balance-sheet rental / ordering-rights extraction / risk warehousing / pass-through), economic counterparty (who is on the other side), base carry driver (what you are fundamentally being paid for), dominant failure driver (what is most likely to cause loss), gross cash-flow source (pre-drag income), main net-return drags (costs and losses), dependency chain flag (whether the payer chain passes through subsidy or reflexive activity), and concentration flags (shared venues, allocators, borrower bases, or order-flow dependencies).
- Adopt a sub-5-minute classification rulebook for practitioners: (1) Split position into holder cash-flow legs; if one row needs two economic sentences, split it. (2) Record immediate payer. (3) Trace at most 2 hops to terminal funding using the five closed labels. (4) Tag one mechanism from the closed list. (5) Record economic counterparty; add adverse incidence annotation only for extractive legs. (6) Tag base carry driver and dominant failure driver. (7) Record gross source and net drags in separate fields. (8) Add fragility overlays only where relevant. (9) Flag dependency chains where the payer is downstream of subsidy or reflexive activity. (10) Aggregate portfolio on the 2D grid of base carry driver × dominant failure driver, always displaying terminal funding source beside each cell.
- Build the primary portfolio dashboard as a 2D grid with base carry driver on one axis and dominant failure driver on the other axis, with each cell showing aggregate exposure and the terminal funding source distribution. Add separate concentration views for: venue/custody concentration, allocator/router concentration, shared borrower base, shared order-flow or builder dependence, and reflexive demand dependence. Use single-axis cuts (by carry driver alone or failure driver alone) only as secondary analytical views, never as the main risk control surface.
- Apply the promotion rule before finalizing any portfolio classification: for every leg where a risk has been classified as a fragility overlay rather than a dominant failure driver, explicitly ask 'Would removing this risk materially compress the expected yield?' If the answer is yes, reclassify it as the dominant failure driver. This should be applied systematically across all seven reference cases (sUSDe, Morpho vault, Uniswap v3 LP, Ethereum MEV, Nexus Mutual, DOVs, Term Finance, Resolv USR) before the framework is considered production-ready.
- Validate the proposed closed vocabularies for base carry driver (inventory-convexity, credit, liquidity-basis, volatility, governance-dilution, ordering-rights access, offchain pass-through) and dominant failure driver (allocator discretion, routing concentration, venue-custody dependence, exit liquidity, endogenous-reflexive demand, crowding, oracle, smart-contract, legal-regulatory, policy-censorship, claims-process) by re-running all seven reference cases through the full ledger schema and confirming no case requires a label outside these lists.
