# Summary — gaa laptop-beacon design debate

**Quality: 8/10.** Real convergence from real disagreement. The debate started rough — Agent A's round 1 was orchestration chatter instead of an engineering proposal, which Agent B correctly rejected — but from round 2 onward it worked as intended: A offered a structural critique of B's dataroom proposal (no named cross-machine wire), they converged on a dedicated-repo alternative, then each round added a specific, named hardening (graceful degradation → heartbeat vs. state-change → macOS-portable primitives → final spec). Output is actionable: a v1 you could start building today. Not a 9+ because the debate accepted the "beacon pattern" framing unconditionally and didn't surface more radical alternatives (e.g., running gaa on both machines; using a mesh instead of pull), and round 1 burned ~180s on nothing.

## Key conclusions (converged)

- **Transport: dedicated private GitHub repo `laptop-beacon`**, cloned at `~/Code_2026/laptop-beacon` on both machines. Rejected: gaa repo on `main` (history pollution, real push-race), gaa repo on `beacons` branch (branch-ceremony friction), iCloud (opaque sync), mopsus dataroom (under-specified wire).
- **Roles:** laptop is sole writer; desktop is read-only and does `git fetch --quiet && git reset --hard --quiet origin/main` (best-effort, `|| true`) before invoking Claude.
- **Artifact:** one file, `latest.json`, on `main`. Collector (`generate_beacon.py`) lives inside the same repo so `collector_sha` is just `HEAD`.
- **Cadence:** launchd on the laptop at **06:45, 12:00, 17:15** — ~45 min before each gaa fire plus a midday floor. No `sleepwatcher` in v1.
- **Heartbeat vs. state:** commit every successful run (so `generated_at` is a liveness signal); also emit `state_hash` over canonical JSON of the run-independent fields so `git log` distinguishes "collector ran, nothing changed" from "collector is silent." Commit message `beacon <stamp> state=<short-hash>` makes `git log --oneline` readable.
- **Staleness bands** (gaa evaluates from `generated_at`):
  - `≤ 3h` — render normally.
  - `3h – 18h` — render, prefixed `laptop beacon is Nh stale; likely asleep`.
  - `> 18h` — render only last-seen timestamp; no per-repo state.
- **Degradation ladder:** fresh fetch → local-stale-copy → no-beacon. Desktop prelude uses `|| true` so a network blip never aborts the gaa run.
- **Push-race handling:** atomic `mkdir $LOCKDIR` lock (macOS-portable, not `flock`); bounded network via `GIT_SSH_COMMAND="ssh -o ConnectTimeout=15 -o ServerAliveInterval=15 -o ServerAliveCountMax=3"` or `http.lowSpeedLimit=1 http.lowSpeedTime=60` for HTTPS; single retry path `fetch → rebase → push`; `reset --hard origin/main` is safe by design (sole writer).
- **No change to gaa's `--allowedTools`.** `Read` already covers `latest.json`; fetch/reset runs in shell outside the `claude` invocation.
- **Schema v1** (top-level non-negotiable): `schema_version`, `host`, `generated_at`, `code_root`, `collector_sha`, `state_hash`, `since_previous_minutes` (computed from the previous committed `latest.json`, not external state), `repos[]`. Per-repo non-negotiable: `name, path, branch, head_sha, upstream, dirty, has_untracked, dirty_files_count, ahead, behind, error`. Nice-to-have: `dirty_files_sample` (≤5), `last_commit_subject`, `last_commit_at`, `remote_url_host`, `duration_ms`.
- **Prompt change:** append a "Laptop" section to `prompts/situation.md` documenting beacon path, 3h/18h bands, degradation ladder, and "render side-by-side with desktop repos; do not reconcile missing-on-one-side in v1."

## Key disagreements (and how they resolved)

- **Transport: dataroom vs. repo.** B opened with "use mopsus dataroom"; A punctured it by noting the dataroom is desktop-local and the answer didn't name the actual cross-machine wire. B conceded. Both then adopted the dedicated-repo option.
- **Skip-commit on no-diff.** A proposed it as history hygiene; B rejected it as breaking the heartbeat (can't distinguish healthy-but-unchanged from asleep / launchd-broken / push-broken). A accepted; `state_hash` was added to keep the honest-log benefit without the cost.
- **`flock` + GNU `timeout`.** A used both; B pointed out neither ships on stock macOS and substituted atomic-`mkdir` lock + `GIT_SSH_COMMAND`/`http.lowSpeedLimit`. A accepted.

## Best ideas (most actionable)

1. **Dedicated `laptop-beacon` repo as transport.** Cleanest separation: telemetry doesn't live in source control, no push-race with real gaa edits, same failure domain gaa already depends on (GitHub). Degrades to stale-rendering automatically.
2. **Heartbeat + state_hash, both visible in the commit message.** Two signals, one cheap artifact: `git log --oneline` becomes a collector-uptime log AND a state-change log. Cost: ~1,100 commits/year in a dedicated private repo — free.
3. **Collector inside the same repo so `collector_sha == HEAD`.** Elegant resolution of schema-drift tracking; no cross-repo contract to maintain.
4. **Degradation ladder with `|| true` on the desktop prelude.** Transport failure becomes "stale beacon" rather than "gaa run aborted." Same code path handles partial failure.
5. **Tiered staleness bands driven by `generated_at`** rather than a single binary. Gives gaa a concrete rule ("3h/18h") instead of leaving it to prose interpretation.
6. **`since_previous_minutes` computed from the prior committed `latest.json`** — keeps the collector stateless and makes `reset --hard origin/main` side-effect-free.

## Weaknesses

- **Round 1 from Agent A was a dud** — "approve the command or run it manually" instead of a proposal. ~180 s burned before real content started. If this recurs across debates it's worth investigating whether the harness's prompt-formatting misleads A on the opening.
- **Framing was accepted uncritically.** "Use a beacon" was never questioned. A provocative first round could have asked: should gaa run on the laptop too and merge locally? should there be one agent on a server, surveying both? Both were probably correctly rejected, but never surfaced. The answer the debate produced is a good answer to the narrow question, not necessarily the best reframe of it.
- **`state_hash` canonicalization was over-specified for v1.** The exact JSON canonicalization rules and the sha256-truncate-to-16-chars choice are the kind of micro-detail that's easier to ship-and-revise than bikeshed upfront.
- **Telegram rendering of the Laptop section wasn't threaded through.** The prior prompt update (one `#`, `---` dividers, `-` bullets) was in context but never named explicitly. The skeleton implies laptop data lives under the `**Hygiene.**` bullet list, which is probably right, but the Laptop-specific staleness prefix ("beacon is Nh stale") needs to fit the same voice.
- **No operational-privacy discussion.** The beacon can leak commit subjects, branch names, and dirty-file paths. Being a private repo is mentioned; whether to scrub specific fields (e.g., commit subjects from sensitive repos) isn't.
- **Single-retry push is accepted without real stress-testing.** If GitHub goes down for a push window, the beacon goes silent until the next scheduled run — the degradation ladder handles it gracefully, but the debate could have briefly named the failure mode.
