# Stress-test the "Couples That Code" v1 product spec

The attached context is a v1 product design for **Couples That Code**: a two-sided marketplace bundling weekend lodging + a facilitated agentic-coding workshop for romantic-partner pairs. One bundled SKU ("Couples Code Camp", Fri–Sun), curated lighthouse properties first then overlay/certified hosts later, founder-taught workshops for the first ~50 weekends, BYOD laptops with venue-provided dual-monitor pair-coding stations, a privacy/secops moat ("your IP stays yours"), and managed-AI tokens billed pass-through with markup as a secondary revenue line.

Debate, hard, the spec's core bets:

1. **Bundled-SKU-only strategy** — is refusing a "stay only" tier brand-defining or growth-suicide?
2. **Founder-taught workshop model** — does it create a tight v1 product, or is it a fragile bottleneck masquerading as discipline?
3. **Curated-then-overlay supply path** — is "lighthouse properties first" the only credible order, or vanity capex that delays product/market fit signals you'd get faster from overlay?
4. **Privacy/secops moat vs. agentic-AI workshop tension** — is the BYO-key + side-project-framing escape hatch enough, or does the spec want to have it both ways?
5. **Wedge realism** — is "couples who both code professionally and want to ship side projects together on weekends" a real population large enough to build a category around, or a delightful-sounding micro-niche that caps at a lifestyle business?

Push for concrete numbers, falsifiable predictions, and named comparable companies (Selina, Hacker House, On Deck, Soho House, Airbnb Experiences, Recurse Center, etc.) where they sharpen the argument. Reviewer should be willing to call the whole thesis a vanity project if warranted; architect should be willing to materially revise the spec if the critique lands.
