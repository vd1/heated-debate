# Debate Summary — Vault Composition Adapter Review

**Quality score: 8/10**

Strong debate with genuine convergence. B (gpt-5.4) landed multiple findings that forced substantive design changes from A (claude-opus-4-6). A was initially too optimistic about compatibility ("no risk engine changes required", "~20 lines for path confidence") — claims that didn't survive source scrutiny. B's sharpest contribution: reframing from graph-expansion to evidence-model problem.

## Key Conclusions (converged)

1. **"No risk engine changes" is indefensible.** Blast radius, opacity, and exit liquidity all treat traversal as unweighted reachability. All three need path-product weighting for partial allocations.
2. **`Edge` needs `EdgeEvidence` list** — multiple provenance records per semantic edge, with weight/confidence resolution rules.
3. **Adapter runs post-BFS with shared state** (visited set, depth map, boundaries) — not BFS re-entry, not inline resolver.
4. **Manifest cycles are valid input** — the deUSD/xUSD cycle IS the motivating incident. Flagged as completeness warning, not rejected.
5. **`composition_ids` = direct manifest targets only** (not all descendants). Separate from `upstream_ids`. Passed to venue adapters for PLEDGE/LEND attachment.
6. **Opacity needs contribution-weighted accounting by proof class** with global mass-conservation cap. Not best-path-per-leaf.
7. **`weight: None` means "unknown"** — treated as 1.0 only for risk-conservative consumption (blast radius), never to improve transparency (opacity).

## Best Ideas

1. `EdgeEvidence` list with operative weight/confidence resolution (Rounds 3-5)
2. Sparse node seeding so `merge_from` lets Enso overwrite (Round 2)
3. Contribution-weighted opacity by proof class with global cap (Rounds 4-5)
4. `composition_ids` as narrow separate eligibility set (Round 5)
5. Conflict rules: highest weight for exposure, lowest weight for transparency (Round 5)

## What the Debate Corrected in the Original Spec

| Original spec claim | Debate correction |
|---|---|
| No risk engine changes needed | 3 of 5 signals need weighted path-product refactor |
| Enso data takes precedence via merge_from | merge_from is first-write-wins; adapter must seed sparse nodes |
| Post-BFS re-entry with own expansion | Post-BFS with shared BFS state object |
| Edge dedup is fine | Same-type dedup silently loses provenance; needs evidence-append |
| Manifest validates and rejects cycles | Cycles are the motivating use case; accept and flag |
| Composition children reuse upstream_ids | Separate composition_ids, direct targets only |

## Key Disagreements (resolved late)

- **Venue eligibility scope:** 3 rounds to converge on "direct manifest targets only, separate set"
- **Weight merge semantics:** A's initial "Enso weight=1.0 overrides manual" was backwards; resolved to "explicit weights only from manifest unless another source also provides one"
- **Manifest cycle handling:** A initially proposed rejection; B correctly pointed out this breaks the motivating use case

## Weaknesses

- A made 3 wrong claims in Round 1 that should have been caught by reading source code
- No performance analysis of path enumeration on production graphs
- The weighted risk engine refactor (~350 lines) is substantially larger than the original "small adapter" framing suggested
