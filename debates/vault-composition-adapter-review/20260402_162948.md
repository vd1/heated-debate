## Vault Composition Adapter — Design Review

### 1. Pipeline Ordering & BFS Re-entry Mechanics

**Sound decisions:**
- Placing the adapter *after* initial BFS but *before* venue adapters is correct — injected children need full downstream expansion, and venue adapters (Morpho, Aave, Euler) need the complete token set including composition children
- Re-entering BFS from injected children with `downstream_depth` remaining hops is the right semantic — they're structurally equivalent to Enso-discovered underlyings

**Concerns:**
- **The spec says "re-enter downstream BFS" but doesn't specify which BFS function is called.** If `_attach_composition_edges` calls back into the downstream loop of `build_neighborhood_double_bfs`, it needs access to the same `visited_ds` set, or it will re-expand nodes already visited. The current BFS uses a local `visited_ds` that's scoped to the `while q_ds` loop and goes out of scope before the adapter runs.
  - **Decision needed:** Either pass the visited set into the adapter, or have the adapter run its own mini-BFS with a *fresh* visited set (acceptable if the graph's `add_node`/`add_edge` dedup handles collisions, which it does via `merge_from` and `_edge_set`)
- **`downstream_depth` parameter is ambiguous.** Is it the *remaining* hops from the opaque node, or the *total* configured depth? If deUSD is discovered at depth 2 of a depth-4 BFS, do its children get 2 remaining hops or 4? The spec says "up to `downstream_depth` remaining hops" — this should be the *remaining* budget, not the global config. But the signature takes a flat `downstream_depth: int` with no information about where in the original BFS the opaque node was found.
  - **Recommendation:** Track the discovery depth of each manifest-matched node and pass `max(0, configured_depth - discovery_depth)` as the re-entry budget
- **No upstream re-entry.** The adapter only re-enters downstream. If a composition child (e.g., xUSD) has *upstream* dependents (wrappers built on top of it), those are missed. This is probably fine for the initial use case (we want to decompose, not discover further wrappers of internals), but should be an explicit design decision, not an omission.

### 2. Manifest Format Sufficiency

**Adequate for v1:**
- Chain-keyed → address-keyed → allocations is clean
- `edge_type` being explicit (CONTAINS vs VAULT) is correct — these have different semantics in the risk engine's flow adjacency
- `weight` with equal-split default is pragmatic
- `source` provenance is well-mapped to opacity classes

**Missing fields / insufficiencies:**
- **No `last_verified` timestamp per allocation.** The top-level `source: "manual:2026-03-15"` dates the whole entry, but individual allocations may be verified at different times (e.g., one allocation confirmed on-chain, another from a multisig snapshot two months ago)
- **No `confidence` or `accuracy` field.** A weight of 0.45 implies precision that manual curation may not support. Consider adding `weight_confidence: "exact" | "estimated" | "unknown"` to feed the risk engine
- **No `chain_id` per allocation target.** If an opaque vault on Ethereum has allocations on Arbitrum (cross-chain strategies), the current format assumes all allocations share the parent's chain. Add optional `chain_id` per allocation entry
- **No versioning or schema version field.** When the format evolves, there's no way to distinguish v1 from v2 manifests
- **No `total_weight` constraint validation.** Weights can sum to >1.0 or <1.0. The adapter should validate and warn (but not reject — partial visibility is better than none)
- **Radical alternative: Consider making the manifest a DAG, not a flat list.** If xUSD is itself opaque and has its own allocations, the current format requires a *separate* top-level entry for xUSD. A nested format (allocations with sub-allocations) would be more natural for recursive vaults, though it trades simplicity for expressiveness.

### 3. Edge Cases

**Recursive opaque vaults (vault A → opaque vault B → opaque vault C):**
- The design handles this *if* all three have manifest entries. The adapter iterates all nodes in `graph.nodes` and matches against `(chain_id, address)`. After injecting A's children (including B), B is now in the graph. But **the adapter has already iterated past B** in a single pass.
- **Critical fix needed:** The adapter must iterate to a fixpoint or use a worklist. One pass is insufficient if manifest entry B depends on manifest entry A having already been injected.
- **Recommendation:** Use a worklist: start with all manifest-matched nodes, inject their children, then check if any *newly added* nodes also match the manifest, repeat until no new matches. Bound by manifest size to guarantee termination.

**Node conflicts:**
- If a composition child already exists in the graph (discovered by Enso with different metadata), `Graph.add_node` calls `merge_from`, which keeps existing non-blank fields and fills blanks from the new node. This is correct — Enso data should take precedence over manifest data for fields Enso provides.
- **Edge conflict:** If Enso already created a WRAPS edge from parent to child, and the manifest says CONTAINS, both edges will exist (different `edge.key` = `(src, type, dst)`). This is semantically wrong — you'd have the same relationship typed two ways. The adapter should check for existing composition edges between the pair and skip or upgrade.

**Manifest cycles (A allocates to B, B allocates to A):**
- The BFS re-entry uses its own visited set, so it won't infinite-loop on the BFS side. But the *worklist* fixpoint approach described above would infinite-loop if A→B→A→B... 
- **Fix:** Track which manifest entries have been applied (the spec already mentions `graph.meta["compositionAdapterEntries"]`). The worklist should check this before re-applying.

**Weight arithmetic edge cases:**
- Weights >1.0 (leveraged vault), weights summing to 0 (degenerate entry), negative weights — all need validation
- Equal-split when `weight` is absent: if one allocation has a weight and another doesn't, what happens? The spec doesn't address mixed explicit/implicit weights.

### 4. Risk Engine Interaction

**Cycle detection:**
- The risk engine's `_build_flow_adj` adds reverse flow for COMPOSITION_EDGES (CONTAINS is in that set). So injected CONTAINS edges from deUSD→xUSD will create a reverse MINT flow xUSD→deUSD. If xUSD is pledged as collateral somewhere and deUSD can be borrowed, the cycle detector *will* find the leverage cycle. This is the primary design goal and it works correctly.
- **However:** the cycle detector requires a PLEDGE edge in the cycle. If the manifest reveals a pure composition cycle (A contains B, B contains A) with no lending market, it won't be flagged as a leverage cycle. This is arguably correct (pure composition cycles are a different class of risk), but the system should still detect and report them separately.

**Blast radius:**
- `blast_radius` walks upstream via COMPOSITION_EDGES from the root, collecting related tokens, then finds PLEDGE edges. Injected composition edges are traversed identically. **No changes needed — confirmed sound.**
- **Nuance:** injected edges with `source: "manual:*"` should arguably contribute to blast radius with a discount factor, since the allocation fractions are attested, not proven. The current blast radius calculation ignores provenance entirely.

**Opacity:**
- This is where the design has its biggest win. `opacity_score` walks downstream through COMPOSITION_EDGES. Without the adapter, an opaque vault is a leaf with no further composition successors → 100% unaccounted. With the adapter, it walks through to the allocation targets → opacity drops.
- **Provenance mapping gap:** The spec maps `manual:*` → `offchain_asserted`. But `opacity_score()` in the current code classifies proof classes by checking `leaf.source == "onchain"` or `leaf.symbol in _TERMINAL_SYMBOLS` or `leaf.tvl is not None`. It does **not** check for `offchain_asserted`. The injected edges carry `source` from the manifest, but the *child nodes* created by the adapter need their `source` field set to something the opacity scorer recognizes.
  - **If the child node is then expanded by BFS re-entry**, its source will be overwritten to `"enso"` by `node_from_enso_payload`. This is actually fine — the node's provenance reflects how it was discovered, while the *edge's* provenance reflects how the composition relationship was established. But the opacity scorer doesn't look at edge sources, only node sources.
  - **Recommendation:** The opacity scorer should consider the *weakest link* in the path — if any edge in the composition path has `source: "manual:*"`, the leaf should be classified as `offchain_asserted` regardless of the node's own source.

### 5. Provenance/Opacity Mapping Correctness

**Current state:**
- The spec's provenance table (`onchain:* → onchain_provable`, `manual:* → offchain_asserted`) is clean but exists only in the design doc. The code needs to actually implement this mapping.
- The opacity scorer in `neigh_risk.py` uses a different classification scheme: `onchain_provable`, `oracle_attested`, `offchain_asserted` — but the classification logic doesn't consume edge-level source fields at all. It only looks at leaf node properties.

**Correctness issues:**
- **Manual composition → Enso-discovered leaf:** If you manually assert deUSD contains xUSD, but xUSD is a normal Enso token with on-chain provable TVL, the *edge* is attested but the *leaf* is provable. What's the correct opacity class? The weakest-link principle says `offchain_asserted`, because the *relationship* is unverified even if the destination is real.
- **Weight accuracy feeds opacity:** If the manifest says 45% allocation to xUSD but the real allocation is 30%, the opacity score will be wrong. Weights should have a staleness/confidence dimension that feeds into the unaccounted fraction.

### 6. Missing Concerns & Gaps

**Manifest governance:**
- Who can edit the manifest? A single malicious or stale entry could suppress risk signals (inject fake composition → opacity drops → risk looks better than reality)
- **Recommendation:** Manifest changes should be PR-gated with mandatory review. Consider signing manifest entries or requiring multi-party attestation.

**Staleness / liveness:**
- The spec's open question #2 asks about staleness warnings. This is critical, not optional. Vault allocations change — deUSD's allocation to xUSD was the problem because the *cycle* was introduced by a reallocation. A manifest entry from March 2025 might be dangerously wrong by November 2025.
- **Recommendation:** Mandatory staleness warnings (configurable threshold, default 30 days). Stale entries should increase opacity, not be treated as fresh.

**Weight drift:**
- Even fresh manual entries have weight drift. A vault that was 45% xUSD yesterday might be 60% today after rebalancing. The manifest is a snapshot, not a live feed.
- **Radical alternative:** Instead of fixed weights, allow the manifest to specify a "weight range" (`weight_min: 0.3, weight_max: 0.6`) and have the risk engine run scenarios at both extremes.

**Multi-hop manifest dependencies:**
- If manifest entry A references address B, and B also has a manifest entry, the order of processing matters (discussed in §3). But there's a deeper issue: the manifest could form a dependency graph that contradicts on-chain data. If Enso says X wraps Y, but the manifest says X contains Z (not Y), which wins? The current design adds both — Enso edge + manifest edge — creating a node with two different composition stories.
- **Recommendation:** Add a conflict detection pass that warns when manifest edges contradict Enso edges for the same parent.

**No upstream cone contamination check:**
- The adapter injects children of opaque nodes. These children get BFS re-entry downstream. But do they also get added to `upstream_ids`? If not, venue adapters won't attach lending markets for composition children. If the adapter runs before venue adapters (which it does), and venue adapters filter by `eligible_ids=upstream_ids`, then composition children discovered by the adapter won't be eligible for venue attachment unless `upstream_ids` is updated.
- **This is a bug in the design.** The adapter needs to either: (a) add injected children to `upstream_ids`, or (b) venue adapters need a broader eligibility set that includes composition-discovered tokens.

**No dry-run / diff mode:**
- For auditing, it would be valuable to run the adapter in dry-run mode that shows *what would be injected* without modifying the graph. This supports manifest review workflows.

**Testing gap:**
- No negative test: what happens when the manifest references an address that Enso returns valid data for (i.e., the vault *isn't* opaque anymore)? The adapter would inject composition edges that compete with Enso's own composition edges. Need a test and handling for this "opacity resolved" case.

## Verdict

**The design is fundamentally sound** — it solves a real and critical gap (the deUSD/xUSD depeg scenario is a compelling motivator). The pipeline position, edge type reuse, and risk engine compatibility are all well-considered.

**Must-fix before implementation (3 items):**
1. **Worklist iteration** instead of single-pass to handle recursive opaque vaults
2. **Update `upstream_ids`** with composition children so venue adapters see them
3. **Weakest-link provenance** in opacity scoring — edge source must influence leaf classification

**Should-fix (4 items):**
- Depth budget tracking (remaining hops, not global config)
- Edge conflict detection (manifest vs. Enso composition edges on same pair)
- Manifest staleness enforcement with opacity penalty
- Weight validation (sum bounds, mixed explicit/implicit handling)

**Nice-to-have (3 items):**
- Per-allocation `chain_id` for cross-chain strategies
- Weight confidence / range for scenario analysis
- Dry-run mode for manifest auditing

**Architectural Challenge**

The current design treats this as a graph-expansion problem. It is really an evidence-model problem. Manual composition is not equivalent to Enso-discovered structure; it is a weaker claim about structure. If you encode both as ordinary composition edges, the risk engine will over-trust asserted relationships.

"Add `weight` to `Edge`" was necessary but insufficient. The actual problem is deeper: **three of five risk signals treat graph traversal as unweighted reachability.**

1. Add `weight` to `Edge`
2. Make `blast_radius` multiply exposure by path-product of weights
3. Make `opacity_score` multiply leaf TVL by path-product of weights

### Finding 2 — Conceded. Opacity needs contribution-weighted accounting, not best-path.

B is right that picking the strongest path per leaf is wrong when weights exist. The correct model:

For each leaf L reachable from root R:
- Enumerate all simple paths P1, P2, ... from R to L
- For each path Pi, compute:
  - `path_weight(Pi) = product(edge.weight for edge in Pi)` (allocation fraction reaching L via this path)
  - `path_confidence(Pi) = min(edge.confidence for edge in Pi)` (weakest link in this path)
- Aggregate per proof class:
  - `verifiable_contribution(L) = sum(path_weight(Pi) * L.tvl for Pi where path_confidence == onchain_provable)`
  - `attested_contribution(L) = sum(path_weight(Pi) * L.tvl for Pi where path_confidence == oracle_attested)`
  - `asserted_contribution(L) = sum(path_weight(Pi) * L.tvl for Pi where path_confidence == offchain_asserted)`

This is not a small scorer tweak. The current `walk()` collapses at the node level, so this needs a path-enumeration traversal. For the graphs we're building (tens to low hundreds of nodes, DAG-like with rare convergence), all-simple-paths is tractable. For pathological cases, cap path count per leaf.

### Finding 3 — Weight resolution rule for conflicting evidence

When `EdgeEvidence[]` has multiple entries with different weights:

**Rule: highest-confidence evidence wins for weight; if tied, use the conservative (lower) weight.**

Concretely:
- If Enso says A→B with no weight (implicit 1.0) and manifest says A→B with weight 0.45:
  - Enso evidence is `confidence: "enso"` (oracle_attested)
  - Manifest evidence is `confidence: "offchain_asserted"`
  - Enso wins → weight 1.0 operative
  - This is correct: if Enso can see the relationship, its implicit full-weight is more trustworthy than a manual fraction
- If two manual entries disagree (weight 0.45 vs weight 0.30):
  - Same confidence tier → conservative wins → weight 0.30 operative
  - Log a warning for manifest conflict

For risk signals:
- `blast_radius`: use operative weight for path-product (conservative = less exposure claimed = safer)
- `opacity_score`: use operative weight for contribution accounting
- `exit_liquidity`: use operative weight for redemption path weighting

**Fallback:** If no evidence has a weight, default to 1.0 (current unweighted behavior — backward compatible).

### Finding 4 — Mass conservation and convergent paths

This is the hardest remaining problem. In a DAG where A→B (weight 0.6) and A→C (weight 0.4), and both B→D (weight 1.0) and C→D (weight 1.0):
- Path A→B→D contributes 0.6 of A's TVL reaching D
- Path A→C→D contributes 0.4 of A's TVL reaching D
- Total: 1.0 of A's TVL attributed to D — correct

But if B→D weight is 0.5 and C→D weight is 0.5:
- Path A→B→D contributes 0.3
- Path A→C→D contributes 0.2
- Total: 0.5 — correct, because 50% of B and C's TVL goes to D

So path-product with sum-of-paths works for DAGs **as long as weights represent allocation fractions and the DAG is acyclic at the manifest level.**

**For cycles:**
- **Manifest cycles (A→B→A) should be rejected at validation time.** A vault cannot be allocated to itself or to a vault that allocates back to it. This is not valid portfolio structure — it's a data entry error or a genuine leverage cycle that should be detected by the risk engine, not asserted by the manifest.
- **Validation rule:** Load manifest as a directed graph, run SCC detection. Any SCC with >1 node is rejected with an error naming the cycle. Single-node entries are fine (no self-loops since `src != dst` by construction).

**For convergent paths in the BFS-expanded graph (not manifest-only):**
- The traversal may discover the same node via both manifest-injected and Enso-discovered paths. Path-product sum can exceed 1.0 if the graph has structural redundancy (e.g., Enso says A wraps D at weight 1.0, and manifest says A contains B at 0.6 which contains D at 1.0 — total 1.6x).
- **Cap rule:** Per-leaf contribution is capped at `min(sum_of_path_products, 1.0) * leaf_tvl`. This prevents over-counting without requiring global flow normalization.
