Review the vault composition adapter design spec for the neigh (token neighborhood graph) system. The adapter fills gaps where Enso API cannot see vault internals (e.g., Elixir deUSD backed by xUSD). It uses a JSON manifest to inject composition edges into the graph, then re-enters BFS for downstream expansion. Debate whether the design is sound, focusing on: 1) Pipeline ordering and BFS re-entry mechanics 2) Manifest format sufficiency 3) Edge cases (recursive opaque vaults, node conflicts, manifest cycles) 4) Risk engine interaction with injected composition edges (cycle detection, blast radius, opacity) 5) Provenance/opacity mapping correctness 6) Missing concerns or gaps.
