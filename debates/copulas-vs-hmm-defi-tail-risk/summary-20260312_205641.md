# Debate Summary: Copulas vs HMMs for DeFi Tail Risk

## Quality Score: 7/10

Strong convergence toward a concrete, implementable spec (v0.3 by round 5). Good intellectual progression: the architect's initial hybrid proposal was correctly challenged as over-parameterized for sparse DeFi data, leading to a genuinely better robust-first architecture. Weakness: both agents are codex instances with similar priors, so there was no deep pushback on whether distributionally robust bounds are actually tractable at per-block latency — the feasibility concern was raised but resolved by fiat ("fallback hierarchy") rather than by working through the math. No real disagreement survived past round 2.

## Key Conclusions

- **Robust bounds first, not copulas or HMMs.** Both agents converged on distributionally robust dependence bounds as the production core, with copula/HMM coupling kept offline until promotion gates pass.
- **Regime model is control-plane only.** A lightweight 2-3 state semi-Markov model drives action throttles (margin, LTV, haircuts), but does NOT feed into joint tail aggregation — avoiding double-counting.
- **Governance as transition/jump risk.** Smart contract exploits, oracle halts, and bridge pauses are handled as deterministic incident states that bypass smoothness budgets on affected pairs.
- **Uncertainty is split.** Dependence ambiguity (`W_dep`) is separated from data-missingness uncertainty (`W_miss`), preventing outages from silently loosening risk controls.
- **Policy monotonicity is formalized.** Worse risk state can never produce looser actions, enforced via CI assertions across all bucket/state combinations.

## Key Disagreements

- **Largely resolved by round 2.** The architect initially proposed a full HMM + regime-copula hybrid; the reviewer argued it was too parameter-heavy for DeFi's data reality. The architect conceded and pivoted to robust-first. No major disagreement persisted after that.
- **Tractability remains hand-waved.** The reviewer flagged that `sup/inf` over a constrained dependence set per block could be expensive. The resolution (M0/M1/M2 fallback hierarchy) is pragmatic but untested.

## Best Ideas

1. **Adversarial dependence search with persistence penalty** — optimize worst-case loss over feasible dependence each hour, but penalize flip-flopping. Directly asks "what dependence hurts most now?" rather than assuming one true copula.
2. **Feature firewall with conditional MI test** — controller features and PD components must be disjoint, tested via conditional mutual information on residuals after market-factor controls. Prevents hidden double-counting.
3. **Bounded fail-closed infeasibility** — when the ambiguity set is infeasible, relax in a fixed order (topology -> tail -> correlation) with capped step sizes, else conservative fallback. No silent degradation.
4. **Crisis twin sandbox** (mentioned but not developed) — agent-based liquidation/bridge/oracle simulator to generate adversarial but structurally plausible dependence paths for stress testing.
5. **Conformal risk bands** on top of robust bounds for finite-sample coverage guarantees — a concrete way to handle the "we don't have enough tail data" problem.

## Weaknesses

- **No real stress-testing of the robust bounds approach itself.** What if the ambiguity set is too conservative and over-throttles utilization? The agents acknowledged this risk but didn't quantify it.
- **Parameter defaults are reasonable but unjustified.** `b_step = P95`, `epsilon = 0.02`, dwell times of 30/60/120 blocks — these are sensible starting points but pulled from intuition, not calibration.
- **Both agents ignored computational cost.** Per-block linearized approximation (M1) was proposed without discussing approximation error bounds or when M1 diverges meaningfully from M0.
- **No discussion of how to bootstrap the ambiguity set bounds** (pairwise correlation ranges, tail dependence ranges) from DeFi's limited history. This is the hardest practical problem and was sidestepped.
