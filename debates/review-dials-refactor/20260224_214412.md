### 🔌 Prefix injection strategy

**pynaille.py mutates stored history in-place** — the creativity meta-instruction gets permanently baked into history. In round 3, Agent A's full context shows round 1's user message saying `[Creativity: 5/5] Explore radical alternatives…` even though we're now converging. This contaminates future context.

shelley.sh avoids this correctly (fresh string each turn, not stored). Fix for pynaille.py:
```python
# Don't mutate; send ephemeral-prefixed copy
last = history_a[-1]
prefixed_msgs = [*history_a[:-1], {"role": last["role"], "content": prefix + last["content"]}]
r_a = call_with_retry(agent_a, prefixed_msgs, temp_a)
```

**Longer-term alternative**: update `history_a[0]["content"]` (system message) per round — keeps meta-instructions cleanly separated from conversation content. Risk: some models don't handle mid-conversation system prompt changes gracefully.

### Agent B · `sonnet` · 56s

Two structural assumptions worth challenging before the implementation settles:

---

**Challenge #1 — Symmetric dials are wrong for a debate**

Both agents run the same dial schedule. At round 1 (dial 5), a "wild" critic invents objections that may not survive scrutiny. At round 5 (dial 1), a "precise" critic stops pushing back just when the architect needs final stress-testing.

The critic's role is inverted from the architect's. Consider role-differentiated schedules:

```python
def get_dial_a(round_num, total_rounds):  # architect: 5→1, explore then converge
    ...

def get_dial_b(round_num, total_rounds):  # critic: stays 2-3, structured pressure
    t = round_num / max(total_rounds - 1, 1)
    return max(2, round(3 - t))  # 3→2, stays critical but grounded
```

Or invert B entirely (1→5): start with precise objections, open up to creative alternatives as A's proposal hardens. Either is more defensible than symmetry.

---

**Challenge #2 — Does the text prefix actually do anything?**

The creativity instruction is the only lever shelley.sh has (no `--temperature` flag in the claude CLI). In pynaille.py, both temperature *and* text prefix change. The question: is the text prefix load-bearing, or is temperature doing all the work?

An empirical test: run pynaille.py with and without prefix injection at identical temperatures. If outputs are indistinguishable, the prefix is cargo-cult and you should drop it from pynaille.py entirely — clean history management for free.

If it *is* load-bearing, shelley.sh is feature-incomplete and needs a workaround (e.g., a proxy that wraps the claude CLI to inject temperature via a model config).

---

**Latent issue — O(n²) token growth**

Both implementations append all history every round. A 10-round debate with 1000 tokens/response sends ~55k tokens total in API calls (round 1: 1k, round 2: 2k, ..., round 10: 10k). For long debates or large topics this will hit context limits or become expensive.

Neither script has a pruning strategy. Options in order of complexity:
- Fixed sliding window (keep last N exchanges)
- Summarize early rounds into a single compressed message
- Separate "working memory" from "full log" — agents get summarized context, log retains full fidelity

Worth a `--max-history-rounds` flag now before the design hardens.

---

**Minor: the priming message in pynaille.py**

```python
history_b.append({"role": "assistant", "content": "Understood. Send me the first proposal."})
```

This fabricated assistant turn exists to prime B's reviewer identity. It's reasonable, but untested: does removing it change B's behavior? If not, it's dead weight that occupies tokens every round. If yes, the system prompt is doing less work than assumed and the priming message is doing more — which suggests the system prompt wording needs strengthening instead.

**Challenge #1 — Symmetric dials**

Agreed that symmetry is lazy, but the inversion proposal needs more scrutiny. The "critic at 1→5" framing sounds compelling until you stress-test it: in round 5, a "wild" critic on a nearly-converged proposal tends to produce hallucinated objections and scope-creep, exactly when you want methodical stress-testing. The problem isn't the direction, it's the coupling to a 5-level scale designed for generative creativity, not adversarial critique.

A cleaner model: critics don't belong on the same dial at all. The architect needs a creativity schedule; the critic needs a *pressure* schedule — how hard to push, not how creative to be. These are orthogonal axes. A dial that runs `stringency: [lenient, moderate, rigorous]` 1→3 for B is more defensible than any mirroring of A's creativity scale.

Near-term pragmatic fix that doesn't require a new abstraction: constant B dial at 2 ("Refine, fix, tighten"). A focused critic throughout is better than a symmetric one.

---

**Challenge #2 — Is the prefix load-bearing?**

The empirical test is the right call, but there's a prior: temperature controls the *distribution* of possible outputs, while the text prefix controls *which region* of that distribution the model samples from. At temperature 1.0, both matter; at 0.3, temperature has already narrowed the distribution so much that the prefix has diminishing marginal effect. This suggests:

- shelley.sh (no temperature control): prefix is the *only* lever — definitely load-bearing
- pynaille.py round 1 (temp=1.0): prefix probably adds signal
- pynaille.py round 5 (temp=0.3): prefix might be redundant

If the test confirms redundancy at low temps, the right fix in pynaille.py isn't to drop the prefix entirely — it's to drop it conditionally at `temp < threshold`, or only inject it into the system prompt via `system=` kwarg (where it's always respected regardless of temperature). The "cargo-cult" framing would be correct for user-message injection at low temp, but not for system-level injection.

The shelley.sh "feature incomplete" point is fair. A pragmatic workaround short of a proxy: write a thin Python wrapper that calls the claude SDK directly with both `--temperature` and prefix, replacing the `claude -p` call. Or accept the asymmetry and document it — pynaille gets the full feature set, shelley.sh is the fast/dirty variant.

---

**Challenge #3 — O(n²) token growth**

The math holds. The impact is worse in shelley.sh because `--resume` doesn't let you control what's in the session — the entire conversation accumulates invisibly and you can't prune it. pynaille.py at least gives you full control of the messages array.

The `--max-history-rounds` flag is the right incremental addition. For implementation: keep a rolling window of the last N exchange pairs in the messages array, but always keep the system message and the original topic. The topic is the most important anchor to preserve — losing it in a sliding window causes drift.

One option worth flagging before the design hardens: separate history management from the debate loop entirely. A `History` class with `.append()`, `.as_messages(max_rounds=N)`, and `.summarize()` methods makes the pruning strategy swappable without touching the loop logic.

---

**Challenge #4 — Priming message**

The fabricated assistant turn is doing something real: it establishes turn structure for models that are sensitive to role alternation. Without it, B's first actual message has `system → user(topic+proposal)` with no prior assistant turn, which some models handle fine and others treat as an unusual context. The 10-token cost is negligible.

The more interesting question is whether the *content* of the priming message matters. `"Understood. Send me the first proposal."` signals passivity — B is waiting to receive, not primed to attack. Changing it to something like `"Ready. I'll focus on flaws, missing cases, and overengineering."` might shift B's default posture more than any dial adjustment. Worth testing before the priming message gets canonized.
