## Topic

Design a Python function that takes a list of stock trades (ticker, qty, price, buy/sell) and outputs a portfolio summary with current positions, average cost basis, and realized P&L. Propose an implementation plan: data structures, function signatures, edge cases.

## Round 1/3

### Agent A (sonnet) — 50s

## Plan: Portfolio Summary Function

### Data Structures
Three dataclasses, clean separation of concerns:

| Class | Purpose |
|---|---|
| `Trade` | Input record — ticker, qty, price, side |
| `Position` | Mutable state per ticker — net_qty, avg_cost, realized_pnl |
| `PortfolioSummary` | Output — open_positions dict, realized P&L by ticker + total |

---

### Function Signatures
```python
def compute_portfolio(trades: list[Trade]) -> PortfolioSummary
def _validate_trade(trade: Trade) -> None  # raises ValueError
```

---

### Algorithm (Average Cost Method)

**Buy:** `new_avg = (old_qty × old_avg + qty × price) / (old_qty + qty)`

**Sell:** `realized_pnl += (sell_price − avg_cost) × qty_sold` — avg_cost of remaining shares is unchanged.

Positions that reach `net_qty == 0` exit `open_positions` but their P&L stays in `realized_pnl_by_ticker`.

---

### Edge Cases

| Case | Behavior |
|---|---|
| `qty ≤ 0` / `price ≤ 0` | `ValueError` |
| Sell > held qty | `ValueError` (`allow_short=False` default) |
| Sell with no position | `ValueError` |
| Full exit | Removed from open positions |
| Empty input | Returns zeroed summary |

---

### File Layout
```
portfolio/
  models.py    # dataclasses
  compute.py   # compute_portfolio(), _validate_trade()
  tests/test_compute.py
```

**Key design decision:** Average cost method (not FIFO) — simpler, common for portfolio tracking. FIFO can be added later via a `lot_queue: deque[tuple[float, float]]` per ticker if tax-lot accuracy is needed.

### Agent B (sonnet) — 52s

## Review

Solid skeleton, but several real flaws need fixing before implementation.

---

### 🔴 Critical

**1. `float` for financial math is wrong.**
`0.1 + 0.2 != 0.3`. Every average-cost calculation compounds the error.

```diff
- qty: float
- price: float
+ from decimal import Decimal
+ qty: Decimal
+ price: Decimal
```

**2. `allow_short` is mentioned in the edge case table but not in the signature.**
This is a contradiction — it's a behavioral parameter that must be explicit.

```diff
- def compute_portfolio(trades: list[Trade]) -> PortfolioSummary
+ def compute_portfolio(
+     trades: list[Trade],
+     allow_short: bool = False,
+ ) -> PortfolioSummary
```

**3. No `timestamp` / ordering guarantee on `Trade`.**
Average-cost is order-dependent. An unsorted input silently produces wrong results. Either enforce sorted input or sort internally — but the plan says nothing.

```diff
+ # Option A: document the contract
+ # trades must be ordered chronologically; raises ValueError otherwise

+ # Option B: add to Trade and sort internally
+ timestamp: datetime
```

---

### 🟡 Design Flaws

**4. `side` should be an `Enum`, not an implied string.**
Nothing in the plan prevents `Trade(side="BUY")`, `Trade(side="buy")`, `Trade(side="b")` from silently diverging.

```diff
+ class Side(Enum):
+     BUY = "buy"
+     SELL = "sell"

- side: str  # implied
+ side: Side
```

**5. `_validate_trade()` is the wrong home for trade validation.**
That logic belongs in `Trade.__post_init__`. A standalone private function can be called zero or multiple times; a dataclass invariant is always enforced.

```diff
- def _validate_trade(trade: Trade) -> None:  # raises ValueError
+ @dataclass
+ class Trade:
+     ...
+     def __post_init__(self):
+         if self.qty <= 0 or self.price <= 0:
+             raise ValueError(...)
```

**6. Ownership of `realized_pnl` is ambiguous.**
`Position` tracks it *and* `PortfolioSummary.realized_pnl_by_ticker` also tracks it. When a position closes and is removed from `open_positions`, how does P&L migrate? The plan doesn't say. Simplest fix: P&L lives only on `Position` (even closed ones), and `PortfolioSummary` aggregates from a single source of truth.

```diff
- open_positions: dict[str, Position]  # closed positions discarded
+ positions: dict[str, Position]        # all tickers, open or closed
+ # Position.net_qty == 0  →  closed; filter at presentation layer
```

---

### 🟠 Missing Edge Cases

| Missing | Risk |
|---|---|
| Ticker case sensitivity — `AAPL` vs `aapl` | Silent duplicate positions |
| Multi-cycle same ticker (open → close → reopen) | P&L accumulation must be additive, not reset |
| Fractional qty (e.g., crypto 0.001 BTC) | Relevant if `qty` is typed as `int` by mistake |
| No current prices → unrealized P&L impossible | `PortfolioSummary` implies completeness it can't deliver |

On the last point: the function name and output promise a "portfolio summary," but without current market prices, unrealized P&L is structurally absent. Either rename to `realized_summary` or accept an optional `current_prices: dict[str, Decimal]` parameter.

---

### Minor

The three-file layout (`models.py`, `compute.py`, `tests/`) is appropriate. The FIFO footnote is a good forward-compatibility note. The algorithm itself (weighted average cost, sell doesn't change avg) is correct.

### Agent A (sonnet) — 50s

Here's the revised plan addressing all six issues:

---

### What changed and why

**1. `Decimal` everywhere** — `float` is wrong for money. `Decimal` is exact and composes correctly through the weighted-average formula.

**2. `allow_short: bool = False` in signature** — Was referenced in edge cases but missing from the API. Now explicit.

**3. `timestamp: datetime` on `Trade`, sorted internally** — Average cost is order-dependent. Rather than documenting a caller contract (easy to violate silently), `compute_portfolio` sorts by timestamp before iterating. Deterministic regardless of input order.

**4. `Side(Enum)` instead of `Literal["buy", "sell"]`** — Prevents `"BUY"`, `"Buy"`, `"b"` from silently falling through. The Enum variant is the only valid value.

**5. Validation moved to `Trade.__post_init__`** — A standalone `_validate_trade()` can be skipped or called twice. A dataclass invariant fires on every construction path, including `dataclasses.replace()`.

**6. Single source of truth for P&L** — `realized_pnl` lives only on `Position`. `PortfolioSummary.positions` holds **all** tickers (open and closed). The `open_positions` property filters on `net_qty != 0` at read time. No migration logic needed on exit, no duplication to get out of sync.

**Bonus fixes from the missing edge cases:**
- Ticker normalization to uppercase in `__post_init__`
- Multi-cycle accumulation explicitly specified (`realized_pnl` additive, `avg_cost` resets on reopen)
- `current_prices: dict[str, Decimal] | None = None` parameter added — without it the function only honestly returns a *realized* summary, not a full portfolio summary

### Agent B (sonnet) — 42s

## Review of Revision

All six issues addressed correctly. Three new problems introduced or left unresolved.

---

### 🔴 New Critical: Float→Decimal at the Construction Boundary

The plan fixes internal arithmetic but ignores the entry point. If callers write:

```python
Trade(ticker="AAPL", qty=100, price=150.5, ...)
```

`Decimal(150.5)` silently produces `Decimal('150.4999999999999715782905695...)`. The fix must be in `__post_init__`:

```python
def __post_init__(self):
    object.__setattr__(self, "price", Decimal(str(self.price)))
    object.__setattr__(self, "qty", Decimal(str(self.qty)))
```

Without this, `Decimal` provides false confidence — the precision loss just moved from arithmetic to ingestion.

---

### 🟡 `current_prices` Keys Need the Same Normalization as Tickers

The plan normalizes `trade.ticker` to uppercase in `__post_init__`. But `current_prices` is a raw `dict[str, Decimal]` passed by the caller. If a caller passes `{"aapl": Decimal("200")}`, it silently finds no match for `"AAPL"`. The function must normalize those keys too:

```python
current_prices = {k.upper(): v for k, v in (current_prices or {}).items()}
```

---

### 🟡 `allow_short=True` Semantics Are Unspecified

The flag is now in the signature, but the plan says nothing about what short positions mean for average cost. A short has inverted P&L: cost basis is the *sale* price, gain comes from covering lower. The current algorithm (buy updates avg_cost, sell computes gain against avg_cost) breaks if `net_qty` goes negative. This needs either:

- An explicit statement that `allow_short=True` is out of scope for this version, or  
- A separate code path: when `net_qty < 0`, track `avg_short_price` and compute P&L on cover buys

Leaving it undocumented means the flag exists but the implementation is silently wrong for its non-default case.

---

### Minor

**Timestamp ties:** Python's sort is stable, so tie resolution follows input order — acceptable, but worth a one-line doc comment so callers understand the behavior.

**Unrealized P&L shape:** `PortfolioSummary` with `current_prices` still has no specified field for unrealized P&L or a total including both realized and unrealized. Given that was a primary motivation for adding the parameter, the output structure should be explicit.

---

### Verdict

The revision is solid. The float→Decimal boundary issue and `current_prices` normalization are implementation bugs waiting to happen; the `allow_short` gap is a design ambiguity. Fix those three and this is ready to implement.

### Agent A (sonnet) — 61s

Rev 3 changes:

---

**1. `Decimal(str(...))` coercion in `__post_init__` (critical)**

`Decimal(150.5)` inherits the float's binary approximation error. `Decimal(str(150.5))` → `Decimal('150.5')` exactly. The fix is in `__post_init__` so it's automatic regardless of whether the caller passes `int`, `float`, or `Decimal`:

```python
object.__setattr__(self, "price", Decimal(str(self.price)))
```

`object.__setattr__` is needed because `@dataclass` with `frozen=False` (default) does allow normal assignment — but using it here avoids any future footgun if the class is made frozen.

**2. `current_prices` key normalization at function entry**

```python
current_prices = {k.upper(): v for k, v in (current_prices or {}).items()}
```

Happens once at the top of `compute_portfolio`, before any lookups.

**3. `allow_short=True` → `NotImplementedError` explicitly**

Short P&L has inverted semantics (cost basis is the sale price, gain on cover buys). Implementing it requires a separate code path. The choice: ship the flag wired to `NotImplementedError` now so the API is forward-compatible, rather than either omitting the parameter or silently producing wrong results when `net_qty` goes negative. Short selling is the obvious v2 candidate.

**4/5. Minor: stable sort documented, unrealized P&L fields made explicit in the dataclass.**

### Agent B (sonnet) — 42s

## Review of Revision 3

The three critical issues are resolved correctly. Three remaining problems.

---

### 🔴 `current_prices` Values Have the Same Float Problem

The fix coerces `Trade.price` but not `current_prices` values. A caller writing:

```python
compute_portfolio(trades, current_prices={"AAPL": 200.5})
```

feeds a float directly into unrealized P&L arithmetic. The normalization step should coerce values too:

```diff
- current_prices = {k.upper(): v for k, v in (current_prices or {}).items()}
+ current_prices = {k.upper(): Decimal(str(v)) for k, v in (current_prices or {}).items()}
```

Same class of bug as the one just fixed, same fix.

---

### 🟡 `object.__setattr__` Explanation Is Inverted

The plan says `object.__setattr__` is used because `frozen=False` "does allow normal assignment." That's backwards — if `frozen=False`, `self.price = ...` works fine and is clearer. `object.__setattr__` is only *required* for frozen dataclasses, where normal assignment raises `FrozenInstanceError`. The code is harmless, but the stated rationale contradicts itself. Either use `self.price = Decimal(str(self.price))` now, or freeze the dataclass and keep `object.__setattr__` with a correct explanation.

---

### 🟡 Partial `current_prices` Coverage Unspecified

If three positions are open and only two have entries in `current_prices`, the third position's unrealized P&L is unknown. The plan doesn't say whether that position gets `None`, `Decimal(0)`, or is omitted from unrealized totals. Whichever choice is made, it needs to be explicit — silent `Decimal(0)` would distort `total_unrealized_pnl`. Recommended: omit from the per-ticker dict and exclude from the total, with a clear doc comment.

---

### Minor

`NotImplementedError` for `allow_short=True` conventionally signals "subclass and override this." For an unfinished feature, `ValueError("short selling not yet supported")` is more appropriate and gives a cleaner error message to callers.

---

### Verdict

One new critical miss (float coercion on `current_prices` values), two design gaps to specify. Otherwise this is clean — structurally sound, semantically correct for the implemented cases. Close.
