## Phase B: Pendle Market Metadata Enrichment — Plan Review

PT/YT/SY tokens already appear in the neighborhood graph as regular nodes via
Enso BFS (role=stripped, edge type=STRIP). But they carry no Pendle-specific
data — no implied APR, no market TVL, no expiry. This phase enriches existing PT
nodes with market-level metadata without adding new nodes or edges.

### Proposed implementation

1. **Refactor shared market fetch (core/neigh_helpers.py)**
   - Extract the paginated fetch loop from `fetch_expired_pendle_addresses()` into `_fetch_all_pendle_markets(chain_id)` with session-level caching
   - `fetch_expired_pendle_addresses()` becomes a thin consumer of the shared fetch
   - No behavior change, just eliminates redundant API calls

2. **New adapter (core/neigh_adapter_pendle.py, ~50 lines)**
   - `_enrich_pendle_pt_nodes(graph, eligible_ids)` — does NOT create nodes or edges
   - Builds pt_address → market lookup from `_fetch_all_pendle_markets()`
   - Walks graph nodes matching PT pattern (symbol starts with "PT-", project=pendle)
   - Injects into node.extra: implied_apr, base_apr, reward_apr, expiry, pendle_market_address
   - Sets node.tvl from market liquidity if not already set
   - Does NOT overwrite apy_base (Pendle implied APR is a different concept)

3. **Wire into BFS and CLI**
   - Add `with_pendle: bool` to `build_neighborhood_double_bfs()`

   - Call after venue adapters with same eligible_ids=upstream_ids

   - PT node enriched with implied_apr/expiry
   - apy_base NOT overwritten
   - YT/SY nodes not enriched
   - No matching market → node unchanged
   - Fetch failure → warning, no crash

## Design Question Rulings

### 1. Enrich only PT nodes, or YT too?

**Ruling: Enrich both PT and YT.** PT-only is wrong.

- A Pendle market has exactly one PT and one YT. The market's `impliedApr` is equally meaningful for both — it's the discount-to-maturity for PT and the yield exposure for YT. They're two sides of the same position.
- YT nodes already have `role="stripped"` and are in the graph. Withholding market data from them is an arbitrary asymmetry.
- The lookup is trivial: build both `pt_address → market` and `yt_address → market` from the same market list.
- **Tradeoff**: ~10 extra lines of code, zero extra API calls.

### 2. Enrich SY nodes with aggregate data?

**Ruling: No. Skip SY enrichment for now.**

- SY is many-to-many with markets (one SY can back multiple PT/YT pairs with different expiries). Aggregating implied APR across expiries is meaningless — you'd be averaging a 2% 30-day implied with a 12% 1-year implied.
- SY's role isn't even `"stripped"` — it falls through to `"wrapper"` or `"lp"` based on underlying count. The graph model doesn't distinguish SY from any other wrapper.
- If SY enrichment becomes needed later, it should be a separate decision: promote SY to its own role first (`role="yield_wrapper"` or similar), then enrich.

### 3. v2 API or v1 active-only?

**Ruling: Stay on v2 (`/core/v2/markets/all`).** But with a caveat.

- v2 is already used by `fetch_expired_pendle_addresses()` and returns all markets (active + expired), which is what we want.
- v1 used by pendleWatch filters `isActive=true` — that's wrong for enrichment because expired-but-in-graph PTs would be missed.
- **Critical caveat**: The v2 market objects contain a `details` dict with `impliedApy`, `base`, `reward`, `liquidity` — the same fields pendleWatch parses via `to_rows()`. But the current `fetch_expired_pendle_addresses` only reads `expiry`/`pt`/`yt`/`sy` and discards everything else. **Verify that v2 response objects include the `details` block before committing.** If v2 strips `details`, the entire enrichment approach falls apart and you'd need to call v1 separately or switch endpoints.
- **Hedge**: Add a debug log in `_fetch_all_pendle_markets` that warns if the first market object lacks a `details` key. Fail-fast rather than silently producing empty enrichment.

### 4. Enrich expired-but-in-graph PTs?

**Ruling: Yes, unconditionally.**

- The v2 API already returns expired markets. The data is free.
- Expired PTs in the graph represent held positions. Their implied APR at expiry (should be ~0), expiry date, and historical TVL are useful for strategy evaluation.
- The adapter doesn't need to distinguish — just match any PT/YT node to its market, expired or not.

### 5. Abstraction boundary: `_fetch_all_pendle_markets` vs higher-level lookup?

**Ruling: Keep `_fetch_all_pendle_markets` returning raw list. Build the lookup in the adapter.**

- The expired-addresses consumer iterates the list differently (tracking SY active/expired state). A lookup dict would be wrong for that use case.
- The enrichment adapter needs `pt_address → market` AND `yt_address → market`. Building these is 5 lines — not worth abstracting.
- If a third consumer appears, promote to a shared `build_pendle_market_lookup(chain_id) → Dict[str, market]` at that point. Not before.

## Challenges to the Premise

### TVL semantics are wrong

The plan says: *"Sets node.tvl from market liquidity if not already set."*

**This conflates two different things:**
- `node.tvl` for token nodes = circulating supply × price (from Enso)
- Market `liquidity` = AMM pool depth in the Pendle market

A PT token with $50M circulating but $2M market depth should not show `tvl=2M`. The Enso-sourced TVL (if present) is more accurate for what `tvl` means on a token node.

**Counter-proposal**: Put market liquidity in `extra.market_liquidity_usd`. Only set `node.tvl` if it's `None` AND you're confident the market liquidity is a reasonable proxy (e.g., for very illiquid PTs where Enso returns nothing). Add a comment explaining the semantic difference.

### The "adapter" naming is misleading

Every other adapter (`_attach_*_market_nodes`) creates new venue nodes + edges. This one mutates existing token nodes in-place. Calling it `neigh_adapter_pendle.py` with function `_enrich_pendle_pt_nodes` signals a different contract.

**Options:**
- **(A)** Name it `neigh_enricher_pendle.py` / `_enrich_pendle_nodes` — explicitly different from adapters
- **(B)** Keep `neigh_adapter_pendle.py` but add a module docstring explaining the enrichment-only pattern
- **(C)** Radical: make it a venue adapter that creates a `pendle_market` node + `STRIP_MARKET` edge, like Morpho/Aave create `lending_market` nodes

**Recommendation: (A).** The naming should encode the behavioral difference. When someone sees `adapter_*` they expect new nodes. When they see `enricher_*` they expect mutation of existing nodes. This is a pattern you may reuse (e.g., enriching Aave aTokens with reserve data from the token side rather than the market side).

### Node matching heuristic is fragile

The plan says: *"symbol starts with 'PT-', project=pendle"*

- `project` field comes from Enso. It may be `None`, `"Pendle"`, `"pendle"`, or some variant.
- `symbol` prefix works for PT (`PT-`) and YT (`YT-`), but Enso could return symbols without the prefix in edge cases.
- The `protocolSlug` field (used in `infer_role_from_enso`) is more reliable than `project`.

**Recommendation**: Match on `role == "stripped"` AND `protocol` contains `"pendle"` (case-insensitive). This reuses the existing role inference rather than re-implementing symbol matching. To distinguish PT from YT for the lookup, THEN check symbol prefix — but only after the role+protocol gate.

### The `_parse_pendle_addr` format difference is a latent bug

v2 API returns token references as `"1-0xabc..."` strings. v1 returns nested dicts. The shared `_fetch_all_pendle_markets` will return v2 market objects. The enrichment adapter will use `_parse_pendle_addr` to extract PT/YT addresses. **This works today.**

But if pendleWatch is ever migrated to consume the shared cache (a natural next step), its `to_rows()` function expects nested dicts, not string IDs. Document this format dependency in the shared fetcher's docstring.

## Radical Alternative (Considered and Rejected)

**Create `pendle_market` venue nodes like other adapters do**, with a `MARKET` edge from PT → pendle_market node. This would:
- Keep PT nodes pure (no mutation)
- Make market data queryable as a first-class entity
- Be consistent with the Aave/Morpho/Euler pattern

**Why rejected:**
- A Pendle market isn't a venue you "deploy" tokens to (unlike lending markets). PT/YT tokens ARE the market. Adding a venue node creates a redundant entity.
- The plan's premise — "the existence of a Pendle market is implicit from the PT node" — is correct.
- Enrichment-in-place is the right pattern for metadata that describes the node itself rather than a relationship.
