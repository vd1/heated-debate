## Topic

> Design a Python function that takes a list of stock trades (ticker, qty, price, buy/sell) and outputs a portfolio summary with current positions, average cost basis, and realized P&L. Propose an implementation plan: data structures, function signatures, edge cases.

## Round 1 of 3 — Initial Plan & First Review

### 🏗️ Agent A · 50s

**Plan: Portfolio Summary Function**

#### Data Structures

Three dataclasses, clean separation of concerns:

| Class | Purpose |
|---|---|
| `Trade` | Input record — ticker, qty, price, side |
| `Position` | Mutable state per ticker — net_qty, avg_cost, realized_pnl |
| `PortfolioSummary` | Output — open_positions dict, realized P&L by ticker + total |

#### Function Signatures

```python
def compute_portfolio(trades: list[Trade]) -> PortfolioSummary
def _validate_trade(trade: Trade) -> None  # raises ValueError
```

#### Algorithm — Average Cost Method

**Buy:** `new_avg = (old_qty × old_avg + qty × price) / (old_qty + qty)`

**Sell:** `realized_pnl += (sell_price − avg_cost) × qty_sold` — avg_cost of remaining shares is unchanged.

Positions that reach `net_qty == 0` exit `open_positions`; their P&L stays in `realized_pnl_by_ticker`.

#### Edge Cases

| Case | Behavior |
|---|---|
| `qty ≤ 0` / `price ≤ 0` | `ValueError` |
| Sell > held qty | `ValueError` (`allow_short=False` default) |
| Sell with no position | `ValueError` |
| Full exit | Removed from `open_positions` |
| Empty input | Returns zeroed summary |

#### File Layout

```
portfolio/
  models.py          # dataclasses
  compute.py         # compute_portfolio(), _validate_trade()
  tests/
    test_compute.py
```

> **Key design decision:** Average cost method (not FIFO) — simpler, common for portfolio tracking. FIFO can be layered on later via `lot_queue: deque[tuple[float, float]]` per ticker for tax-lot accuracy.

### 🔍 Agent B · 52s

Solid skeleton, but several real flaws need fixing before implementation.

#### 🔴 Critical

**1. `float` for financial math is wrong.**

`0.1 + 0.2 != 0.3`. Every average-cost calculation compounds the error.

```diff
- qty: float
- price: float
+ from decimal import Decimal
+ qty: Decimal
+ price: Decimal
```

**2. `allow_short` referenced in edge cases but absent from the signature.**

A behavioral parameter that affects control flow must be explicit.

```diff
- def compute_portfolio(trades: list[Trade]) -> PortfolioSummary
+ def compute_portfolio(
+     trades: list[Trade],
+     allow_short: bool = False,
+ ) -> PortfolioSummary
```

**3. No `timestamp` / ordering guarantee on `Trade`.**

Average-cost is order-dependent. Unsorted input silently produces wrong results.

```diff
+ # Option A — document the caller contract:
+ # trades must be ordered chronologically; behaviour undefined otherwise

+ # Option B — add field and sort internally:
+ timestamp: datetime
```

#### 🟡 Design Flaws

**4. `side` should be an `Enum`, not an implied string.**

```diff
+ class Side(Enum):
+     BUY = "buy"
+     SELL = "sell"

- side: str  # implied
+ side: Side
```

**5. `_validate_trade()` is the wrong home for validation.**

A standalone private function can be called zero or multiple times. A `__post_init__` invariant fires on every construction path, including `dataclasses.replace()`.

```diff
- def _validate_trade(trade: Trade) -> None:  # raises ValueError
+ @dataclass
+ class Trade:
+     ...
+     def __post_init__(self):
+         if self.qty <= 0 or self.price <= 0:
+             raise ValueError(...)
```

**6. Ownership of `realized_pnl` is ambiguous.**

`Position` tracks it *and* `PortfolioSummary.realized_pnl_by_ticker` also tracks it — two sources of truth that can diverge when a position closes. Simplest fix: P&L lives only on `Position` (even closed ones), aggregated once at read time.

```diff
- open_positions: dict[str, Position]  # closed positions discarded
+ positions: dict[str, Position]       # all tickers, open or closed
+ # net_qty == 0 → closed; filter at presentation layer
```

#### 🟠 Missing Edge Cases

| Missing | Risk |
|---|---|
| Ticker case — `AAPL` vs `aapl` | Silent duplicate positions |
| Multi-cycle same ticker (open → close → reopen) | P&L must accumulate additively, not reset |
| Fractional qty (`0.001 BTC`) | Wrong if `qty` typed as `int` |
| No current prices → unrealized P&L impossible | `PortfolioSummary` implies completeness it can't deliver |

The last point is structural: without market prices, unrealized P&L is absent. Either rename to `realized_summary` or accept `current_prices: dict[str, Decimal] | None`.

#### Minor

File layout is appropriate. FIFO footnote is a good forward-compatibility note. The algorithm itself (weighted average cost; sell doesn't change avg_cost) is correct.

### 🔍 Agent B · 42s

All six issues addressed correctly. Three new problems introduced or left unresolved.

#### 🔴 New Critical: Float→Decimal at the Construction Boundary

The plan fixes internal arithmetic but ignores ingestion. Callers will write:

```python
Trade(ticker="AAPL", qty=100, price=150.5, ...)
```

`Decimal(150.5)` silently produces `Decimal('150.499999971578...')`. The fix belongs in `__post_init__`:

```python
def __post_init__(self):
    object.__setattr__(self, "price", Decimal(str(self.price)))
    object.__setattr__(self, "qty",   Decimal(str(self.qty)))
```

Without this, `Decimal` provides false confidence — precision loss moved from arithmetic to ingestion.

#### 🟡 `current_prices` Keys Need the Same Normalization

Ticker is normalized to uppercase in `Trade.__post_init__`, but `current_prices` keys aren't touched. `{"aapl": Decimal("200")}` silently misses `"AAPL"`. Fix at function entry:

```python
current_prices = {k.upper(): v for k, v in (current_prices or {}).items()}
```

#### 🟡 `allow_short=True` Semantics Are Unspecified

The flag is in the signature, but the plan says nothing about what short positions mean for average cost. Short P&L is inverted: cost basis is the *sale* price, gain comes from covering lower. The current algorithm breaks when `net_qty < 0`. This needs either:

- An explicit statement that `allow_short=True` is out of scope for v1, or
- A separate code path tracking `avg_short_price`, computing P&L on cover buys

Leaving it undocumented means the flag exists but the implementation is silently wrong for its non-default case.

#### Minor

- **Timestamp ties:** Python's sort is stable, so tie resolution follows input order — acceptable, but worth a one-line doc comment.
- **Unrealized P&L shape:** `PortfolioSummary` still has no specified fields for unrealized P&L or a combined total. That was the primary motivation for adding `current_prices`.

> **Verdict:** The revision is solid. The float→Decimal boundary issue and `current_prices` normalization are implementation bugs waiting to happen; `allow_short` is a design ambiguity. Fix those three and this is ready to implement.

**On `Decimal(str(...))`:**
`Decimal(150.5)` inherits the float's binary approximation. `Decimal(str(150.5))` → `Decimal('150.5')` exactly. Automatic for `int`, `float`, or `Decimal` callers.

**On `allow_short=True` → `NotImplementedError`:**
Short P&L requires a separate code path (`avg_short_price`, P&L computed on cover buys). Shipping the flag now as `NotImplementedError` keeps the API forward-compatible without silently wrong results. Short selling is the obvious v2 candidate.

### 🔍 Agent B · 42s

Three critical issues resolved correctly. Three remaining problems.

#### 🔴 `current_prices` Values Have the Same Float Problem

Keys are normalized, but values aren't coerced. A caller writing:

```python
compute_portfolio(trades, current_prices={"AAPL": 200.5})
```

feeds a float directly into unrealized P&L arithmetic. Same class of bug, same fix:

```diff
- current_prices = {k.upper(): v              for k, v in (current_prices or {}).items()}
+ current_prices = {k.upper(): Decimal(str(v)) for k, v in (current_prices or {}).items()}
```

#### 🟡 `object.__setattr__` Explanation Is Inverted

The plan justifies `object.__setattr__` by saying `frozen=False` "allows normal assignment." That's backward — `frozen=False` means `self.price = ...` works fine and is clearer. `object.__setattr__` is only *required* for frozen dataclasses. The code is harmless, but the stated rationale contradicts itself. Fix: either use direct assignment now, or freeze the dataclass and keep `object.__setattr__` with a correct explanation.

#### 🟡 Partial `current_prices` Coverage Unspecified

If three positions are open and only two appear in `current_prices`, the third position's unrealized P&L is unknown. The plan doesn't say whether it gets `None`, `Decimal(0)`, or is omitted from totals. Silent `Decimal(0)` would distort `total_unrealized_pnl`. **Recommended:** omit the position from unrealized totals; document the behavior.

#### Minor

`NotImplementedError` conventionally signals "subclass and override." For an unimplemented feature flag, `ValueError("short selling not yet supported")` is clearer to callers.

> **Verdict:** One new critical miss (float coercion on `current_prices` values), two design gaps to specify. Otherwise structurally sound and semantically correct for the implemented cases. Close.
